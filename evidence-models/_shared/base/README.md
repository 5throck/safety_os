# Evidence Models — Base Schemas

These JSON Schema files are the **single source of truth (SSOT)** for all audit evidence records in Safety OS.

## Schemas

| File | Title | Description |
|------|-------|-------------|
| `finding.schema.json` | Safety Finding | An audit finding: hazard, non-conformance, near-miss, or incident |
| `corrective-action.schema.json` | Corrective Action | A corrective action record linked back to a finding |

## Schema Versioning

Every schema carries a top-level `"version"` field following **semver** (MAJOR.MINOR.PATCH).

| Change type | Version bump | Migration script required? |
|-------------|-------------|---------------------------|
| New optional field | PATCH | No |
| New required field / renamed field | MINOR | Yes |
| Removed field / changed type / new `enum` value | MAJOR | Yes |

Migration scripts live in `evidence-models/migrations/` and are named `<schema>-v<FROM>-to-v<TO>.ts`.

## Phase B Activation Note

Agents **do not write** to evidence-models files in **Phase A** (prototype).
Evidence writes are gated behind Phase B promotion. Until that promotion is confirmed:

- Agents may **read** schemas for validation reference.
- Agents must **not** create or modify `.json` data files in `evidence-models/`.
- All evidence data in Phase A is generated by the `safety-audit.ts` script only.

Phase B activation requires sign-off from the PM (CSO) agent and a passing `bun scripts/safety-audit.ts` run with zero errors.
